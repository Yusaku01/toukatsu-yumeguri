# 東葛湯めぐりマップ 基本設計

## 1. コンセプト

「千葉県東葛地域で日帰り温浴施設を探すときに、地図と一覧からすぐ候補を見つけられる」ことを目的にする。

初期版では、口コミ投稿や会員機能などは持たせず、施設情報を静的データとして管理する。これにより、開発・運用・ホスティングのコストを小さく保つ。

## 2. ターゲットユーザー

主なターゲットユーザー:

- 東葛地域に住んでいる人
- エンジニア
- 首都圏在住の人

利用端末はモバイル主体としつつ、Desktopでも快適に使えるようにする。初期設計では、スマホで地図を開いて近くの候補をすぐ確認できることを優先する。

## 3. 対象エリア

初期対象は千葉県東葛地域とする。

掲載対象は、一般利用できる日帰り温浴施設を中心にする。スーパー銭湯、日帰り温泉、浴槽を主目的にしたサウナ併設施設を含める。宿泊者専用や会員制の浴場は原則含めない。

候補:

- 松戸市
- 柏市
- 流山市
- 我孫子市
- 野田市
- 鎌ケ谷市

必要に応じて、市川市・船橋市など周辺エリアを含めるかは後で決める。

MVPでは、吉川市や船橋市などの周辺施設も初期掲載対象に含める。東葛地域を中心にしつつ、近隣で利用しやすい日帰り温浴施設も扱う。

市区町村フィルターは、データに存在する市区町村だけを表示する。先頭には `すべて` チップを置き、未選択状態やフィルター解除が分かるようにする。

## 4. 主要画面

### 作り方のルール

UIイメージ・参考は、主に以下を見る。

- Google Maps
- Airbnb

Google Mapsからは、地図を主役にした画面構成、検索ボックス、地図上のチップ型フィルター、ピン選択時の施設カード、モバイルでのボトムシート表現を参考にする。

Airbnbからは、モバイルでの検索体験、カードの情報整理、写真や特徴タグの見せ方、地図とリストを行き来する導線、余白とタイポグラフィの扱いを参考にする。

ただし、どちらかをそのまま再現するのではなく、東葛湯めぐりマップに必要な要素だけを残す。MVPでは、市区町村フィルター、現在地、施設カード、公式サイト・外部地図リンクを優先し、ナビゲーションや装飾は増やしすぎない。

UI画像や具体的な画面パターンを集めるときは、Lazywebのプラグイン・skillを使う。特に `lazyweb-quick-references` でGoogle Maps風の地図UI、Airbnb風のモバイル検索・カード・地図/リスト切り替えの参考画像を取得し、実装前に見比べる。

### 地図画面

最優先画面。施設の位置が直感的に分かることを重視する。

UIのざっくりした方向性はGoogle Mapsのように、地図を画面の主役にして、その上に必要最小限の操作だけを重ねる。完全に同じUIにはせず、このサービスでは要素数を絞って軽くする。

モバイル主体のため、片手操作でも市区町村フィルター・現在地・施設カード確認がしやすい配置にする。Desktopでは地図の広さを活かしつつ、施設カードとリスト導線を見つけやすくする。

表示するもの:

- 温浴施設のピン
- 施設名
- 市区町村
- 営業状況または営業時間の要約
- 詳細情報への導線
- 市区町村フィルター
- リスト表示への切り替え導線
- 現在地から探す導線

初期版では、ピンを押したときに簡単な施設カードを表示する。

初期版の画面イメージ:

- 地図はファーストビューの大部分を占める
- 初期表示は松戸・柏・流山・我孫子・野田・鎌ケ谷を含む東葛地域全体が見える範囲にする
- トップ画面は地図主役 + 簡易リスト構成にする
- 上部に市区町村フィルターを置く
- フィルターはチップ状にして、地図上部に横並びで置く
- 左側の大きなサイドメニューは最初は作らない
- スマホでは、選択中の施設カードを画面下部のボトムシート風に表示する
- スマホでは、ボトムシート内で選択中施設と簡易リストを行き来できるようにする
- PCでは、選択中の施設カードや簡易リストを左側または左下に浮かせて表示する
- ズームボタン、現在地ボタンなどは地図ライブラリ標準UIを基本にする
- 地図以外の装飾は抑え、施設名・場所・営業時間・公式リンクの見やすさを優先する

MVPから現在地機能を入れる。ユーザーが許可した場合は現在地付近へ地図を移動し、許可拒否・取得失敗・ブラウザ非対応の場合は、地図表示自体は壊さず、手動で市区町村フィルターを使える状態を保つ。

現在地取得後は、簡易リストと `/list` の一覧を現在地から近い順に並べ、距離も表示する。距離表示は1km未満をm、1km以上をkmで表示する。距離計算や並び替えはコアロジックとして扱い、TDDで振る舞いを確認する。

Desktopでは、左側に簡易リスト/選択施設カードのパネルを置く。パネルは折りたたみ可能にするが、状態は保存せず、初期状態では開いた状態にする。モバイルでは、画面下部に低めのシンプルなボトムシートを表示し、まずはドラッグなしでも成立する実装にする。

施設カードに載せる情報:

- 施設名
- 市区町村
- 住所
- 営業時間
- 定休日
- 確認日
- 公式サイトURL
- Google Maps等の外部地図へのリンク
- メモ、注意事項

MVPの施設カードは、最小実用セットとして上記を表示する。サウナ、露天風呂、駐車場、駅徒歩圏内などの特徴情報は、初期データでは持てる場合だけ保持し、カード上の主要表示や比較UIにはまだ使わない。

営業時間・定休日・メモなど任意項目が空欄の場合、MVPの施設カードではその項目を非表示にする。確認日は表示し、掲載情報がいつ確認されたものかは分かるようにする。

営業中/営業時間外の自動判定はMVPでは行わない。営業時間テキストと確認日だけを表示し、最新情報は公式サイトやGoogle Mapsで確認してもらう。料金情報と施設写真もMVPでは扱わない。施設写真は将来検討とし、初期データ構造には `image` を持たせない。

### リスト画面

地図に出している施設と同じデータを一覧で見られる画面。

MVPでは `/list` を独立ページとして作る。地図が苦手な人や、Desktopで施設情報を比較したい人が使いやすいようにする。

表示するもの:

- 施設名
- 市区町村
- 住所
- 営業時間の要約
- 公式サイトリンク
- 地図で見るリンク

`/list` はカード一覧にする。市区町村フィルターを置き、トップと同じく `すべて` + データに存在する市区町村チップを表示する。現在地取得後は近い順に並べ、現在地未取得時は市区町村順 + 施設名順にする。

初期版の絞り込み候補:

- 市区町村
- サウナあり
- 露天風呂あり
- 駐車場あり
- 駅から徒歩圏内

ただし、最初から絞り込みを作り込みすぎず、まずは市区町村フィルター程度から始める。

MVPでは、市区町村フィルターだけを実装する。サウナ、露天風呂、駐車場、駅から徒歩圏内などの特徴フィルターは、施設データが増えて確認精度を保てる段階で追加する。

施設名・住所のキーワード検索はMVPでは後回しにする。施設数が増え、市区町村フィルターだけでは探しづらくなった段階で追加する。

### 施設詳細

初期版では独立ページではなく、地図画面やリスト画面内の詳細表示で十分。

情報量が増えたら、後から `/spots/[id]` のような施設詳細ページを追加する。

MVPでは施設詳細ページは作らない。施設ごとの情報は、地図上の施設カードとリスト表示で確認できるようにする。将来の詳細ページ追加に備えて、施設データの `id` は安定したslugとして持つ。

### 利用規約・ポリシー

最低限、以下を明記する。

MVPでは `/terms` の1ページに、利用規約、免責事項、外部リンク、プライバシーポリシーをまとめる。`/privacy` は分けず、初期版では運用を軽くする。

- 掲載情報は正確性を保証しないこと
- 営業時間・料金・休館日などは公式サイト等で確認してほしいこと
- 外部サイトへのリンクについて
- 問い合わせ先として作成者のXアカウント `https://x.com/sakku_04` を載せること
- アクセス解析を使う場合は、その利用目的
- 個人情報を原則取得しないこと

初期版ではフォームやログインを作らない想定なので、プライバシーポリシーはかなり軽くできる。

MVPではアクセス解析を入れる。Cloudflare Workers Static Assetsでの配信と相性がよいCloudflare Web Analyticsを第一候補にし、利用目的と取得する情報の概要を `/terms` に明記する。

## 6. データ設計

施設データはJSONで管理し、Astro Content Collection経由で読み込む。

想定配置:

- `src/content.config.ts` に `spas` collectionを定義する
- `src/content/spas/` 配下に温浴施設データのJSONを置く
- 1施設1JSONを基本にし、差分確認とPull Requestレビューをしやすくする
- 拡張子はAstro Content Collectionが標準で扱える `.json` にする。JSONCは標準形式としては使わない
- Zodスキーマで必須項目やURL、緯度経度などを検証する

初期データは、作成者がMarkdown表で手動共有する情報を正とする。Webからの自動収集や自動更新は初期版では行わない。

初期データ共有用のMarkdown表は、以下の項目を基本にする。緯度経度はGoogle Maps URLから抽出・補完する前提にし、手動共有時の必須項目にはしない。

| 施設名 | 市区町村 | 住所 | 公式サイトURL | Google Maps URL | 営業時間 | 定休日 | 確認日 | メモ |
| ------ | -------- | ---- | ------------- | --------------- | -------- | ------ | ------ | ---- |

初期投入時の必須項目:

- 施設名
- 市区町村
- 住所
- Google Maps URL
- 確認日

公式サイトURL、営業時間、定休日、メモは任意とし、不明な場合は空欄を許容する。

初期9件については、Google Maps URLに加えて住所と公式サイトURLも用意する。住所と公式サイトURLは実装時に公式サイトやGoogle Maps表示情報を確認して補完する。確認日は、実装時にこちらが住所や公式サイトURLを確認した日を `lastCheckedAt` として入れる。営業時間・定休日は無理に初期入力せず、空欄の場合は表示しない。

公開リポジトリで扱うため、内部メモ用の `internalNotes` は持たせない。公開表示してよい `notes` だけを使う。

例:

```json
{
  "id": "example-spa",
  "name": "施設名",
  "city": "柏市",
  "address": "千葉県柏市...",
  "lat": 35.0,
  "lng": 139.0,
  "officialUrl": "https://example.com",
  "mapUrl": "https://maps.google.com/...",
  "openingHours": "10:00-24:00",
  "regularHoliday": "不定休",
  "features": {
    "sauna": true,
    "openAirBath": true,
    "parking": true,
    "nearStation": false
  },
  "notes": "営業時間・料金は公式サイトを確認してください。"
}
```

初期版で持つ項目:

- `id`
- `name`
- `city`
- `address`
- `lat`
- `lng`
- `officialUrl`
- `mapUrl`
- `openingHours`
- `regularHoliday`
- `features`
- `notes`

`id` は、こちらで手動設定する英数字slugにする。施設名からの完全自動生成や連番にはせず、データ差分や将来の詳細ページURLで意味が読み取れることを優先する。

市区町村の標準並び順は、松戸市、柏市、流山市、我孫子市、野田市、鎌ケ谷市を優先順にする。未定義の周辺市区町村は名前順で並べる。

後で追加できる項目:

- 最寄り駅
- 駐車場台数
- サウナ温度
- 水風呂
- 岩盤浴
- 食事処
- 料金
- 施設写真
- 最終更新日
- 情報提供者

## 7. 技術方針

できる限りシンプルかつ安く運用する。

開発では、可読性、保守性、単純性、テスト容易性、堅牢性を特に重視する。あわせて、可搬性、拡張性、再利用性、疎結合、高凝集、安全性、性能、互換性、観測性、冪等性も意識する。

原則として、DRY、KISS、YAGNI、Separation of Concerns、Fail Fast、Convention over Configurationを基本にする。SOLIDは必要になった箇所へ自然に取り入れる。

ロジックが複雑、またはサービスのコアになりそうな箇所は、CodexのTDDスキル方針に沿って進める。テストは実装詳細ではなく、公開インターフェースから見える振る舞いを検証する。

テスト環境はVitestを第一候補にする。施設データの検証、Google Maps URLからの緯度経度抽出、距離計算、市区町村フィルターなどを小さくテストする。

E2E/ブラウザテストはMVP初期では必須にしない。まずは手動確認とVitest中心で進め、地図UIやボトムシート、現在地失敗時表示などの重要導線が固まった段階でPlaywrightを追加する。

HTML、CSS、クライアントサイドJavaScript、UI、画像最適化、パフォーマンスに関わる実装前には、`modern-web-guidance` skillを使い、最新のWeb標準・ベストプラクティスを確認する。

初期版のおすすめ方針:

- 静的サイトとして作る
- 施設データはリポジトリ内のJSONで管理し、Astro Content Collection経由で読み込む
- ホスティングはCloudflare Workers Static Assetsを中心に考える
- サーバー、DB、ログイン、投稿機能は最初は作らない

ホスティング方針:

- 第一候補はCloudflare Workers Static Assets
- 静的アセット配信を基本にし、必要になった場合だけWorkers側の処理を足す
- GitHub PagesやVercelは代替候補として扱う
- 初期版ではDBや独自APIを持たず、静的生成したファイルを配信する
- 独自ドメインはMVPでは後回しにし、まずCloudflare WorkersのURLで公開する

採用する技術スタック:

- Astro v6 + 静的生成
- Leaflet + OpenStreetMap
- 素のTypeScript
- Astro Content Collection

地図ライブラリ候補:

- Leaflet + OpenStreetMap
- MapLibre
- Google Maps JavaScript API

低コスト重視なら、初期版はLeaflet + OpenStreetMapが有力。Google Mapsは便利だが、APIキーや課金設定を意識する必要がある。

MVPでは、Leaflet + OpenStreetMapを素のTypeScriptで扱う。Reactは最初から入れず、Astroの静的サイトを基本にしながら、地図部分だけ必要最小限のクライアントJSにする。

地図タイルはMVPではOpenStreetMap標準タイルから始める。アクセスが増えた場合は、利用規約や負荷を見て管理されたタイルサービスへの移行を検討する。ピンはfavicon/温泉ピンの赤茶・橙系を参考にした、ブランド寄せのSVGアセットをLeafletのカスタムアイコンとして使う。

Astroはv6系を使う。Node.jsはAstro v6の推奨に合わせ、Node 22.12.0以上のNode 22 LTS系を前提にする。パッケージマネージャはpnpmにする。TypeScriptはstrict寄りにし、ESLintはAstro/TypeScript向けの最小構成 + Prettier連携から始める。Prettierは標準寄りにし、必要なAstroプラグインだけを入れる。

Cloudflare Workers Static Assetsへのデプロイは、productionのみを対象にし、Cloudflare WorkersのGit integrationでmainブランチへのpushを自動デプロイする。まずは静的出力 + Workers Static Assetsを基本にし、必要になった場合だけCloudflare adapterを採用する。
