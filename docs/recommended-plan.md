# 東葛湯めぐりマップ おすすめ案

## 結論

初期版は、Astro + Leaflet + OpenStreetMap + JSONデータ + Cloudflare Workers Static Assetsで作るのがおすすめ。

現時点では実装に入る前の設計段階として、プロジェクトの目的、ターゲットユーザー、利用端末、画面構成、技術方針を固める。

理由:

- サーバーやDBなしで始められる
- ホスティング費用をほぼ無料にしやすい
- 施設情報をGitHub上のデータとして管理できる
- 地図機能だけ必要なところにJavaScriptを使える
- 後からデザインを変えやすい
- コントリビュート導線を作りやすい

## 前提

### ターゲットユーザー

- 東葛地域に住んでいる人
- エンジニア
- 首都圏在住の人

掲載範囲は東葛地域を中心にしつつ、吉川市や船橋市など近隣で利用しやすい日帰り温浴施設も周辺施設として含める。

### 利用端末

モバイル主体で考える。Desktopにも対応するが、最初に優先するのはスマホで地図を開き、近くの温浴施設をすぐ探せる体験。

エンジニア向けには、GitHub上で施設データや設計意図を追いやすくし、情報追加・修正に参加しやすい構成にする。

## 技術スタック

### 開発方針

開発では、以下を基本姿勢にする。

- 可読性: 未来の自分や他人が読める名前・構造にする
- 保守性: 変更時の影響範囲を小さくする
- 単純性: 必要以上に賢くしすぎず、YAGNIを守る
- テスト容易性: コアロジックを外から検証しやすい形にする
- 堅牢性: 空データ、異常入力、外部リンク切れ、地図ライブラリ読み込み失敗などを考慮する
- 可搬性: OSや環境に依存しすぎない
- 拡張性: 施設データ項目やフィルター追加に備え、自然に拡張できる形にする
- 再利用性: 汎用処理は必要になった時点で共通化する
- 疎結合・高凝集: データ処理、UI表示、地図操作を混ぜすぎない
- 安全性: 外部データやURLを扱う箇所では入力・表示の安全性を意識する
- 性能: モバイルで軽く動くように、不要なJSや重い描画を避ける
- 互換性: 公開後のURLやデータ形式を不用意に壊さない
- 観測性: ビルド・デプロイ・エラー調査をしやすくする
- 冪等性: データ生成やデプロイ処理を何度実行しても破綻しないようにする

有名な原則として、DRY、KISS、YAGNI、Separation of Concerns、Fail Fast、Convention over Configurationを意識する。SOLIDは必要に応じて取り入れるが、初期版では過度な抽象化を避ける。

特に優先するのは、可読性、保守性、単純性、テスト容易性、堅牢性の5つ。

### TDD方針

ロジックが複雑、またはサービスのコアになりそうな箇所は、CodexのTDDスキル方針に従って進める。

TDD対象の候補:

- 施設データの正規化・検証
- Google Maps URLからの緯度経度抽出・補完
- 現在地から各施設までの距離計算と近い順の並び替え
- 市区町村や特徴タグによる絞り込み
- 検索文字列による施設検索
- Google Maps URLや公式URLなど外部リンクの組み立て・検証
- 将来追加するデータ変換、OGP生成、サイトマップ生成などのビルド時処理

進め方:

- 実装前に、公開インターフェースと検証したい振る舞いを決める
- 1つの振る舞いにつき、1つのテストから始める
- Red -> Green -> Refactorを小さく回す
- テストは実装詳細ではなく、公開インターフェースから見える振る舞いを検証する
- まだ必要ない抽象化や網羅しすぎるテストは避ける

テスト環境はVitestを第一候補にする。施設データの検証、Google Maps URLからの緯度経度抽出、距離計算、市区町村フィルターなどを小さくテストする。

E2E/ブラウザテストはMVP初期では必須にしない。まずは手動確認とVitest中心で進め、地図UIやボトムシート、現在地失敗時表示などの重要導線が固まった段階でPlaywrightを追加する。

### Modern Web Guidance

HTML、CSS、クライアントサイドJavaScript、UIコンポーネント、フォーム、画像最適化、地図UI、モバイル表示、パフォーマンスに関わる実装前には、`modern-web-guidance` skillを使って最新のWeb標準・ベストプラクティスを確認する。

特に確認したいテーマ:

- モバイル主体のレイアウト
- 地図上に重ねる市区町村フィルター・現在地UI
- ボトムシートやダイアログ相当のアクセシビリティ
- 画像・OGP・アイコンの読み込み最適化
- LCP、INP、不要なJavaScript削減
- レスポンシブUIとタッチ操作

### フレームワーク

Astro v6系を使う。

このアプリは、地図・リスト・規約ページが中心で、ログインやリアルタイム更新は不要。全体をReactアプリにするより、静的サイトとして軽く作れるAstroが合う。

Reactを使いたくなった場合も、Astroの中で地図コンポーネントだけReactにできる。

Node.jsはAstro v6の推奨に合わせ、Node 22.12.0以上のNode 22 LTS系を前提にする。パッケージマネージャはpnpmにする。TypeScriptはstrict寄りにする。

### 地図

Leaflet + OpenStreetMapを使う。

Google Maps APIは便利だが、APIキーや課金設定が必要になる。初期版では、低コストで扱いやすいLeafletの方がよい。

ただし、施設ごとの外部リンクとしてGoogle MapsのURLは載せる。ユーザーが経路検索したいときはGoogle Mapsに飛ばす。

MVPでは、Leafletを素のTypeScriptで扱う。Reactは最初から入れず、Astroの静的サイトを基本にしながら、地図部分だけ必要最小限のクライアントJSにする。

地図タイルはOpenStreetMap標準タイルから始める。アクセスが増えた場合は、利用規約や負荷を見て管理されたタイルサービスへの移行を検討する。

ピンはfavicon/温泉ピンの赤茶・橙系を参考にしたブランド寄せのカスタムピンにする。形式はSVGアセットにし、Leafletのカスタムアイコンとして扱う。

### データ管理

施設情報はJSONで管理し、Astro Content Collection経由で読み込む。

GitHubでPull Requestを受ける前提なら、最初はJSONで十分。Astro Content Collectionを使うことで、Zodスキーマによる検証、TypeScriptの型安全性、エディタ補完を得られる。人間が編集しやすいことを優先するならYAMLも候補だが、実装のシンプルさではJSONが強い。

想定配置:

- `src/content.config.ts` に `spas` collectionを定義する
- `src/content/spas/` 配下に温浴施設データのJSONを置く
- 1施設1JSONを基本にし、差分確認とPull Requestレビューをしやすくする
- 拡張子はAstro Content Collectionが標準で扱える `.json` にする。JSONCは標準形式としては使わない
- 内部メモ用の `internalNotes` は持たせず、公開表示してよい `notes` だけを使う

初期データは、作成者がMarkdown表で手動共有する情報を正とする。Webからの自動収集、自動更新、スクレイピングは初期版では行わない。

手動共有の基本項目。緯度経度はGoogle Maps URLから抽出・補完する前提にし、手動共有時の必須項目にはしない。

| 施設名 | 市区町村 | 住所 | 公式サイトURL | Google Maps URL | 営業時間 | 定休日 | 確認日 | メモ |
| ------ | -------- | ---- | ------------- | --------------- | -------- | ------ | ------ | ---- |

初期投入時の必須項目は、施設名、市区町村、住所、Google Maps URL、確認日とする。公式サイトURL、営業時間、定休日、メモは任意とし、不明な場合は空欄を許容する。

初期9件については、Google Maps URLに加えて住所と公式サイトURLも用意する。住所と公式サイトURLは実装時に公式サイトやGoogle Maps表示情報を確認して補完する。確認日は、実装時にこちらが住所や公式サイトURLを確認した日を `lastCheckedAt` として入れる。営業時間・定休日は無理に初期入力せず、空欄の場合は表示しない。

おすすめの初期データ項目:

- `id`
- `name`
- `city`
- `address`
- `lat`
- `lng`
- `officialUrl`
- `googleMapsUrl`
- `openingHours`
- `regularHoliday`
- `features`
- `lastCheckedAt`
- `notes`

`id` は、こちらで手動設定する英数字slugにする。施設名からの完全自動生成や連番にはせず、データ差分や将来の詳細ページURLで意味が読み取れることを優先する。

`lastCheckedAt` は入れておいた方がよい。温浴施設の営業時間・料金・休館日は変わりやすいので、いつ確認した情報なのかを表示できる。

料金情報はMVPでは扱わない。営業中/営業時間外の自動判定も行わず、営業時間テキストと確認日を表示し、最新情報は公式サイトやGoogle Mapsで確認してもらう。施設写真は将来検討とし、MVPのデータ構造には `image` を持たせない。

### ホスティング

Cloudflare Workers Static Assetsを中心にする。

おすすめ順:

1. Cloudflare Workers Static Assets
2. Cloudflare Pages
3. GitHub Pages
4. Vercel

初期版は静的生成したファイルをCloudflare Workers Static Assetsで配信する。DBや独自APIは持たず、施設データはビルド時にJSONから読み込む。後から動的な処理が必要になった場合は、同じCloudflare Workers側に小さく処理を足す。

Cloudflare Pages、GitHub Pages、Vercelは代替候補として残す。GitHub PagesはGitHubだけで完結するのが強く、Vercelは使いやすいが、この用途では少し高機能すぎる。

まずはAstroの静的出力 + Cloudflare Workers Static Assetsを基本にする。Cloudflare固有の機能や公式推奨上必要な場合だけ、`@astrojs/cloudflare` などのCloudflare adapterを採用する。

独自ドメインはMVPでは後回しにし、まずCloudflare WorkersのURLで公開する。

### CI/CD

CI/CDが進めやすいように、初期整備として以下を用意する。

- ライセンスはMITにする
- `.github/ISSUE_TEMPLATE/` に `施設追加` と `施設情報修正` の最低限のIssueテンプレートを作る
- `.github/pull_request_template.md` は、変更内容、確認したこと、関連Issueだけの最低限チェックリストにする
- `CONTRIBUTING.md` を用意し、施設情報の追加・修正方法を簡単に書く
- Prettierを入れてフォーマットを統一する
- ESLintを入れて基本的な品質チェックをする
- CIでは、フォーマットチェック、Lint、テスト、ビルドを実行する
- 初期CIのテストはVitest中心にし、PlaywrightはUIが固まってから追加する
- Cloudflare Workers Static AssetsへのproductionデプロイはCloudflare WorkersのGit integrationで実行する
- preview環境はMVPでは作らず、productionのみを対象にする
- mainブランチへのpushでCIが通ったらproductionへ自動デプロイする
- CIでは `pnpm install --frozen-lockfile` を使う
- ESLintはAstro/TypeScript向けの最小構成 + Prettier連携から始める
- Prettierは標準寄りにし、必要なAstroプラグインだけを入れる

## 画面構成

### `/`

地図画面をトップにする。

このサービスで一番価値があるのは「近くにどの施設があるか」をすぐ見ることなので、トップページは地図でよい。

地図の初期表示は、松戸・柏・流山・我孫子・野田・鎌ケ谷を含む東葛地域全体が見える範囲にする。現在地が取得できた場合だけ、ユーザー位置付近へ地図を移動する。

表示するもの:

- 地図
- 施設ピン
- 選択中の施設カード
- 市区町村フィルター
- 現在地ボタン
- リスト画面へのリンク

トップ画面は、地図主役 + 簡易リスト構成にする。レイアウトはGoogle Mapsのように、地図を全面に近い形で表示し、その上に市区町村フィルター・現在地ボタン・リスト切り替えだけを軽く重ねる。Google Mapsほど多機能にはせず、初期版では温浴施設探しに必要な要素へ絞る。

市区町村フィルターは、データに存在する市区町村だけを表示する。先頭に `すべて` チップを置く。市区町村の標準並び順は、松戸市、柏市、流山市、我孫子市、野田市、鎌ケ谷市を優先順にし、未定義の周辺市区町村は名前順にする。

PCでは、上部に市区町村チップを置き、簡易リスト/選択施設カードの左側パネルを表示する。左側パネルは折りたたみ可能にするが、状態は保存せず、初期状態では開いた状態にする。スマホでは、地図上部に市区町村フィルター、画面下部に低めのシンプルなボトムシートを表示し、選択中施設カードと簡易リストを行き来できるようにする。まずはドラッグなしでも成立する実装にする。

MVPから現在地機能を入れる。ユーザーが許可した場合は現在地付近へ地図を移動し、許可拒否・取得失敗・ブラウザ非対応の場合は、地図表示自体は壊さず、市区町村フィルターで探せる状態を保つ。

現在地取得後は、簡易リストと `/list` の一覧を現在地から近い順に並べ、距離も表示する。距離表示は1km未満をm、1km以上をkmで表示する。距離計算や並び替えはコアロジックとして扱い、TDDで振る舞いを確認する。

### `/list`

施設一覧ページ。

MVPで独立ページとして作る。地図が苦手な人や、Desktopで施設情報を比較したい人が使いやすいようにする。

表示形式はカード一覧にする。市区町村フィルターを置き、トップと同じく `すべて` + データに存在する市区町村チップを表示する。現在地取得後は近い順に並べ、現在地未取得時は市区町村順 + 施設名順にする。

表示するもの:

- 施設名
- 市区町村
- 住所
- 営業時間
- 定休日
- 確認日
- 特徴タグ
- 公式サイトリンク
- Google Mapsリンク

MVPでは市区町村フィルターだけを実装する。サウナ・露天風呂・駐車場などの特徴フィルターは、施設データが増えて確認精度を保てる段階で追加する。

施設名・住所のキーワード検索はMVPでは後回しにする。施設数が増え、市区町村フィルターだけでは探しづらくなった段階で追加する。

施設カードとリストのMVP表示は、施設名、市区町村、住所、営業時間、定休日、確認日、公式サイトリンク、Google Mapsリンク、メモを基本にする。特徴タグはデータとして持てる場合だけ保持し、主要表示や比較UIにはまだ使わない。

営業時間・定休日・メモなど任意項目が空欄の場合、MVPではその項目を非表示にする。確認日は表示し、掲載情報がいつ確認されたものかは分かるようにする。

公式サイトやGoogle Mapsなどの外部リンクは、新しいタブで開く。実装時は `target="_blank"` と `rel="noopener noreferrer"` を付ける。

### `/terms`

利用規約とプライバシーポリシーをまとめたページ。

MVPでは `/terms` の1ページに、利用規約、免責事項、外部リンク、プライバシーポリシーをまとめる。`/privacy` は分けず、初期版では運用を軽くする。

書く内容:

- 掲載情報の正確性は保証しない
- 最新情報は公式サイト等で確認してもらう
- 外部リンク先の内容には責任を負わない
- 個人情報は原則取得しない
- Cloudflare Web Analyticsなどのアクセス解析を使うこと
- アクセス解析の利用目的と取得する情報の概要
- 問い合わせ先として作成者X `https://x.com/sakku_04` を載せる

情報追加・修正依頼の主導線はGitHub Issuesにする。作成者Xはプロフィール・補助的な連絡先として載せるが、施設情報の追加・修正はできるだけIssueに集約する。

## デザイン方針

初期版はかなりシンプルでよい。

### UI参考

UIイメージ・参考は以下を主に見る。

- Google Maps
- Airbnb

Google Mapsは、地図を主役にしたレイアウト、検索ボックス、地図上のチップ型フィルター、ピン選択時のカード、モバイルのボトムシート表現の参考にする。

Airbnbは、モバイルでの検索体験、カードの情報密度、特徴タグ、地図とリストを行き来する導線、余白とタイポグラフィの参考にする。

UI画像や具体例を集めるときは、Lazywebのプラグイン・skillを使う。`lazyweb-quick-references` でGoogle Maps風の地図UI、Airbnb風のモバイル検索・カード・地図/リスト切り替えを集め、実装前に見比べる。

ただし、Google MapsやAirbnbをそのまま再現するのではなく、東葛湯めぐりマップでは温浴施設探しに必要な要素だけを残す。

方向性:

- 温かいミニマル和風を基本にする
- ベージュ、赤茶、橙、濃茶をベースにする
- 地図を主役にする
- Google Maps風の、地図上に操作UIを重ねる構成にする
- Airbnb風の、読みやすいカードとモバイル検索体験を参考にする
- 操作UIは市区町村フィルター、現在地、リスト切り替え程度に絞る
- 施設カードは情報を読みやすくする
- 色は多用しない
- 装飾よりも、施設名・場所・営業時間・公式リンクの見やすさを優先する

後から好みのデザインに変えやすいように、初期CSSは薄くする。

CSSはAstroコンポーネント内CSS + global少しで始める。TailwindやSCSSはMVPでは入れない。色、余白、角丸などの最低限のデザイントークンはCSS変数で持つ。カードの角丸は8px前後の控えめな角丸にし、ボタン/チップは少し丸める。地図上のUIチップは白背景 + 薄い影にし、選択状態だけ赤茶/橙のアクセントを使う。

フォントは、本文にNoto Sans JPを使う。サービス名と大見出しだけ少し和風の見出しフォントを使う。Astro v6のFonts APIでパフォーマンス、セルフホスト、フォールバック最適化を担保できる場合に採用し、難しい場合はシステムフォントへフォールバックする。

UIアイコンはlucide由来のものを使う。ただしReactは入れず、必要箇所だけSVGとして利用する。

PWA対応はMVPに入れる。manifestと基本設定を用意し、オフライン対応は最小限にする。地図タイルや現在地はオンライン前提とし、必要なら基本アセットだけキャッシュする。
